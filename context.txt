PHASE 2 — POST-COMPLETION FIXES & DEBUGGING LOG
================================================================================

ISSUE 1 — Wrong Firebase Project in .env.local
  - Root cause: .env.local was pointing to old project moodify-4dba9, Firebase
    console was set up on new project moodify-s2403
  - Symptom: auth/api-key-not-valid error on every Firebase call (signup + Google)
  - Fix: Updated frontend/.env.local with correct config from moodify-s2403
  - Status: FIXED

ISSUE 2 — serviceAccountKey.json from wrong project
  - Root cause: Key was generated from moodify-4dba9, backend pointed to wrong project
  - Blocker: Organization policy (iam.disableServiceAccountKeyCreation) prevented
    generating a new key from moodify-s2403
  - Resolution: Policy was eventually resolved, new key generated from moodify-s2403
  - Fix: Replaced backend/serviceAccountKey.json with new key (project_id: moodify-s2403)
  - Status: FIXED

ISSUE 3 — Dashboard was blank after login
  - Root cause: dashboard/page.tsx was a placeholder returning null
  - Fix: Added temporary dashboard with user display name/email + Sign Out button
  - Note: Will be replaced with full dashboard UI in Phase 5
  - Status: FIXED (temporary)

ISSUE 4 — Password field had no show/hide toggle
  - Fix: Added eye toggle button (lucide-react Eye/EyeOff icons) to password
    input in both LoginForm.tsx and signup/page.tsx
  - Package added: lucide-react
  - Status: FIXED

ISSUE 5 — Token expiry kicking users out after 1 hour
  - Root cause: Cookie set with max-age=3600, token never refreshed
  - Fix: AuthContext now runs setInterval every 55 minutes to force-refresh
    the Firebase token (getIdToken(true)) and update the cookie
  - On auth state change: getIdToken() — no force, reads existing token instantly
  - On interval tick: getIdToken(true) — force refresh only when actually needed
  - Interval is cleared on auth state change and on unmount
  - Status: FIXED

ISSUE 7 — Slow redirect after sign-in
  - Root cause: getIdToken(true) was being called on every auth state change
    (including fresh logins), making an unnecessary network call to Google
  - Fix: Separated initial token read (getIdToken()) from interval refresh
    (getIdToken(true)) — initial read is now instant, no network call
  - Status: FIXED

ISSUE 6 — Google Sign-In errors were silent
  - Root cause: GoogleSignInButton only did console.error, no UI feedback
  - Fix: Added error state + visible red error message below the button
  - Status: FIXED

DECISION — signInWithEmail does NOT call createUserProfile
  - Firestore profile is only created on signUpWithEmail and signInWithGoogle
  - Login via email assumes profile already exists (created at signup)
  - Rationale: Mandatory signup flow — user must register before they can login
